1. Введение
Бизнес-кейс: Данная лабораторная работа имитирует разработку аналитического ядра для стриминговой платформы (аналог Netflix или Кинопоиска). Цель — научиться хранить и обрабатывать разнородные данные, используя наиболее подходящие для каждого типа данных базы данных: каталог фильмов и профили пользователей в MongoDB, логи активности в Cassandra и граф знаний в GraphDB для построения рекомендаций.

Выбранный стек технологий:

MongoDB (Документо-ориентированная): Выбрана для хранения каталога фильмов (movies_meta) из-за гибкости схемы. Это позволяет легко хранить метаданные с изменяющимся набором полей и вложенными структурами (массивы жанров, актеров), что идеально подходит для описания видеоконтента.

Cassandra (Колоночная): Используется для записи потоковых логов действий пользователей. Ее архитектура обеспечивает высокую скорость записи и отличную масштабируемость, что критически важно для обработки миллионов событий просмотра в реальном времени.

GraphDB (Графовая RDF): Применяется для анализа связей между сущностями (актеры, режиссеры, фильмы, жанры). Мощный язык запросов SPARQL позволяет эффективно находить сложные взаимосвязи, необходимые для рекомендательной системы (например, "найти фильмы, где играли те же актеры, что и в фильме X").

3.1.2. Cassandra: Работа с логами просмотров
Обоснование модели данных: Таблица logs спроектирована для максимально быстрой записи. Ключом партиционирования выбран user_id, а ключом кластеризации — timestamp. Это позволяет эффективно получать историю действий конкретного пользователя, отсортированную по времени. Добавление TTL (Time To Live) обеспечивает автоматическое устаревание и удаление старых записей.

Создание Keyspace и таблицы:
Подключаемся к Cassandra через cqlsh (например, docker exec -it <container_id> cqlsh).

sql
-- Создание keyspace с коэффициентом репликации 1 для разработки
CREATE KEYSPACE IF NOT EXISTS streaming_ks
WITH replication = {'class': 'SimpleStrategy', 'replication_factor': 1};

USE streaming_ks;

-- Создание таблицы логов
CREATE TABLE logs (
    user_id UUID,
    timestamp TIMESTAMP,
    action TEXT,
    movie_id INT,
    device_type TEXT,  -- Поле добавлено позже (ALTER TABLE)
    PRIMARY KEY (user_id, timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC);
Изменение схемы таблицы (ALTER TABLE):
Согласно заданию, необходимо добавить поле device_type.

sql
ALTER TABLE logs ADD device_type TEXT;
Вставка данных с TTL:
Здесь мы вставляем записи, которые будут автоматически помечены к удалению через 60 секунд (для демонстрации).

sql

INSERT INTO logs (user_id, timestamp, action, movie_id, device_type) VALUES (uuid(), toTimestamp(now()), 'view', 2, 'tv') USING TTL 60;
INSERT INTO logs (user_id, timestamp, action, movie_id, device_type) VALUES (uuid(), toTimestamp(now()), 'pause', 3, 'tablet') USING TTL 60;

Чтение данных:

sql

Через 60 секунд эти записи перестанут отображаться в выборке, так как они истекут. Это демонстрирует работу механизма TTL для автоматического управления жизненным циклом данных, что полезно для хранения непостоянных данных, таких как история активности за последний час.

3.2. Задание 2 (GraphDB) : Анализ графа знаний с помощью SPARQL
Подготовка: В веб-интерфейсе GraphDB был создан репозиторий movies_repo и в него загружен RDF-датасет о фильмах (например, подмножество данных из LinkedMDB или DBpedia).

SPARQL-запросы для варианта 12:

Найти все фантастические фильмы 80-х годов.
Этот запрос демонстрирует фильтрацию по жанру и периоду времени.

sparql
PREFIX dbo: <http://dbpedia.org/ontology/>
PREFIX dct: <http://purl.org/dc/terms/>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>

SELECT DISTINCT ?film ?titre ?annee
WHERE {
  ?film a dbo:Film ;
        dct:title ?titre ;
        dbo:releaseDate ?date .
  BIND(YEAR(?date) AS ?annee)
  ?film dbo:genre ?genre .
  ?genre rdfs:label "science fiction"@ru . # Или @en, в зависимости от данных
  FILTER(?annee >= 1980 && ?annee < 1990)
}
ORDER BY ?annee
Найти все фильмы, в которых снимались и Роберт Дауни-младший, и Крис Хемсворт.
Этот запрос иллюстрирует поиск фильмов по пересечению актерского состава, что является классической задачей для графовой БД.

sparql
PREFIX dbo: <http://dbpedia.org/ontology/>
PREFIX dct: <http://purl.org/dc/terms/>
PREFIX foaf: <http://xmlns.com/foaf/0.1/>

SELECT DISTINCT ?film ?titre
WHERE {
  ?film a dbo:Film ;
        dct:title ?titre ;
        dbo:starring ?acteur1, ?acteur2 .
  ?acteur1 foaf:name "Роберт Дауни-младший"@ru . # Или Robert Downey Jr.@en
  ?acteur2 foaf:name "Крис Хемсворт"@ru .      # Или Chris Hemsworth@en
  FILTER(?acteur1 != ?acteur2)
}
Ожидаемый результат: Фильмы серии "Мстители" (Avengers).

Вывод:

Сравнение показывает, что количественная популярность (измеряемая числом комментариев) у фильмов эпохи Marvel на порядок выше, чем у классической фантастики 80-х.

Это не означает, что старые фильмы хуже. Это говорит о нескольких факторах:

Масштабирование аудитории: В 80-е аудитория была ограничена кинотеатрами и ТВ. Сейчас, благодаря стриминговым платформам и соцсетям, обсуждаемость и охват зрителей неизмеримо выросли.

Маркетинг и франшизность: Marvel создала огромную взаимосвязанную вселенную, что стимулирует постоянный интерес и обсуждения каждой новой части. Фильмы 80-х, за редким исключением, были отдельными проектами.

Бизнес-вывод: Для современной стриминговой платформы инвестиции в создание или покупку прав на контент, основанный на популярных франшизах (как Marvel) или с участием современных звезд первой величины, с гораздо большей вероятностью приведут к высокому уровню вовлеченности (комментарии, лайки, повторные просмотры), чем вложения в ретро-контент, несмотря на его неоспоримую культурную ценность. Ретро-контент важен для удовлетворения потребностей нишевой аудитории и ценителей, но не является главным драйвером виральной популярности.

4. Заключение: Преимущества полиглотного подхода
Данная лабораторная работа наглядно продемонстрировала эффективность использования полиглотной персистентности. Использование одной универсальной базы данных (например, реляционной) для всех задач привело бы к серьезным проблемам с производительностью и сложностью разработки.

GraphDB (Граф) vs. Остальные: Для выполнения Задания 2 (поиск фильмов по сложным связям) в SQL или MongoDB потребовались бы многоступенчатые запросы с множественными JOIN или операциями $lookup, что крайне неэффективно на больших данных. В GraphDB эта задача решается элегантно и быстро благодаря нативной поддержке графовых структур и языка SPARQL.

Cassandra (Логи) vs. MongoDB: MongoDB также может хранить логи, но она не оптимизирована для таких высоких нагрузок на запись, как Cassandra. Cassandra, благодаря своей архитектуре без master-узлов и линейной масштабируемости, справляется с потоками данных в реальном времени. Возможность задания TTL на уровне таблицы — это "нативная" функция для работы с временными рядами, тогда как в MongoDB чистить старые логи пришлось бы вручную.

MongoDB (Каталог) vs. Cassandra: Cassandra менее гибка в плане схемы: изменение структуры таблицы (ALTER TABLE) — тяжелая операция. Для каталога фильмов, где метаданные могут часто меняться и расширяться (добавили поле trailer_url, потом rating_mpaa), гораздо лучше подходит гибкая схема MongoDB.

Итог: Разделение ответственности между базами данных позволило использовать сильные стороны каждой из них: скорость записи Cassandra, гибкость схем MongoDB и мощь анализа связей GraphDB. Это классический пример архитектуры, где система строится из лучших компонентов для решения конкретных задач, что в итоге обеспечивает высокую производительность, масштабируемость и удобство разработки всего приложения.
